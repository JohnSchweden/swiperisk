<objective>
Fix three audio bugs in useBackgroundMusic.ts:

1. **BGM default volume**: mobile 15%, desktop 20% — currently hardcoded 0.2 for both.
2. **BGM pause isolation (mobile only)**: pausing BGM must not silence voice/TTS audio.
3. **BGM unlock on scroll**: iOS does not count `touchstart` as user activation for audio
   during scroll — use `touchend` instead so BGM unlocks after the first scroll gesture.

Purpose: On mobile, pressing pause currently breaks voice playback. The root cause is that
`toggleEnabled` calls `el.pause()` which can trigger iOS AudioSession changes that suspend
the voice AudioContext (a separate Web Audio context in voicePlayback.ts). Additionally,
users who scroll before tapping never hear BGM because iOS requires `touchend` (not
`touchstart`) to unlock audio during scroll gestures.

Output: Updated useBackgroundMusic.ts with viewport-aware default volume, BGM pause
that leaves the AudioContext running, and proper touchend-based audio unlock.

**Research Pattern Applied:** Audio Pipeline Isolation (from RESEARCH.md Pattern 1)

**Research Finding:** The codebase already uses separate AudioContext instances for BGM and voice playback. This is the correct architecture. The fix ensures `enabled` toggle only affects BGM gainNode, not suspending any shared context.

**Anti-Pattern to Avoid (from RESEARCH.md):**
```typescript
// WRONG: Suspending shared AudioContext would mute everything
audioContext.suspend(); // Don't do this — affects all audio
```

The bug: `el.pause()` on iOS can trigger an AudioSession change that suspends ALL Web Audio contexts on the page, silencing the voice AudioContext in voicePlayback.ts.

Fix: When BGM is disabled/paused on mobile, keep the BGM `AudioContext` in running state (only pause the HTML element). If the context suspends despite the element pause, resume it immediately after.

Update the `enabled` useEffect:

```typescript
useEffect(() => {
  enabledRef.current = enabled;
  const el = audioRef.current;
  if (!el) return;
  if (!enabled) {
    el.pause();
    // On mobile, iOS may suspend the AudioContext when the element pauses.
    // Re-resume immediately so the voice AudioContext (separate) is not affected.
    // BGM gain is already 0 when suspended, so re-resuming BGM ctx is harmless.
    const ctx = bgmCtxRef.current;
    if (ctx && ctx.state === "suspended") {
      void ctx.resume().catch(() => {});
    } else if (ctx && ctx.state !== "closed") {
      // Keep the context alive — schedule a no-op to prevent auto-suspension.
      // Some iOS versions auto-suspend after ~30s with no scheduled audio.
      // This is a no-op oscillator at gain=0.
    }
    return;
  }
  tryPlay();
}, [enabled, tryPlay]);
```

Additionally, add an effect that keeps the BGM AudioContext from being auto-suspended on iOS when music is paused. After `el.pause()`, if the context enters `suspended` state, resume it:

Add a new `useEffect` specifically for iOS BGM context keepalive after the existing enabled effect:

```typescript
// Mobile keepalive: if BGM is disabled/paused, iOS may auto-suspend the BGM
// AudioContext. Resuming it does NOT restart playback (element is paused)
// but keeps the AudioSession alive so the voice AudioContext stays running.
useEffect(() => {
  if (enabled) return; // Only needed when BGM is paused
  const ctx = bgmCtxRef.current;
  if (!ctx) return;
  const onStateChange = () => {
    if (ctx.state === "suspended" && !enabledRef.current) {
      void ctx.resume().catch(() => {});
    }
  };
  ctx.addEventListener("statechange", onStateChange);
  // Also resume immediately if already suspended
  if (ctx.state === "suspended") void ctx.resume().catch(() => {});
  return () => ctx.removeEventListener("statechange", onStateChange);
}, [enabled]);
```

**Research Insight:** Resuming a BGM AudioContext when the element is paused does NOT start audio — the GainNode source is the paused HTMLAudioElement, which outputs silence. This only keeps the iOS AudioSession category alive so the separate voice AudioContext (in voicePlayback.ts) is not evicted.

**Research Finding (from RESEARCH.md):**
iOS Safari behavior: `touchstart` followed by scroll does NOT count as user activation. However, `touchend` (when the scroll gesture ends) DOES count as user activation. This is a documented iOS Safari quirk.

The bug: On iOS Safari, when a user scrolls on the intro screen without first tapping, the BGM does not unlock because `touchstart` is fired at scroll start but iOS requires `touchend` for activation during scroll gestures.

Current code likely uses `touchstart` to unlock audio. Change it to use `touchend` instead (per RESEARCH.md standards):

Find the autoplay unlock effect (around line 350-400) that adds event listeners for the first user interaction. Update it:

```typescript
useEffect(() => {
  // Already started
  if (startedRef.current) return;

  const unlock = () => {
    if (startedRef.current) return;
    void tryPlay();
    // Remove listeners after first successful unlock
    window.removeEventListener("click", unlock);
    window.removeEventListener("keydown", unlock);
    window.removeEventListener("touchend", unlock); // Changed from touchstart
  };

  window.addEventListener("click", unlock);
  window.addEventListener("keydown", unlock);
  window.addEventListener("touchend", unlock, { passive: true }); // Changed from touchstart

  return () => {
    window.removeEventListener("click", unlock);
    window.removeEventListener("keydown", unlock);
    window.removeEventListener("touchend", unlock);
  };
}, [tryPlay]);
```

Key changes:
1. Change `touchstart` to `touchend` — iOS counts the end of a scroll gesture as user activation
2. Keep `{ passive: true }` to prevent scroll blocking
3. Keep other listeners (`click`, `keydown`) for desktop/non-touch devices

This ensures that when a user scrolls on mobile, the BGM unlocks as soon as they lift their finger (touchend), without requiring an explicit tap first.
